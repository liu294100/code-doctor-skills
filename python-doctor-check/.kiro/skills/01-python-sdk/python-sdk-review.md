# Python SDK 代码审查 Skill

当用户打开或提交 Python SDK 项目文件时，按照以下维度进行全面审查。

---

## 一、代码规范与可读性

### 1.1 PEP 8 / Ruff 规范
- 缩进使用 4 空格，禁止 Tab
- 单行不超过 88 字符（Black 默认）或 120 字符（团队约定）
- import 顺序：标准库 → 第三方库 → 本地模块，各组之间空一行
- 禁止 `from module import *`
- 禁止未使用的 import 和变量
- 函数/方法之间空两行（模块级），类内方法之间空一行
- 运算符两侧加空格，逗号后加空格

### 1.2 命名审查
- 类名是否使用 PascalCase
- 函数/变量是否使用 snake_case
- 常量是否使用 UPPER_SNAKE_CASE
- 私有成员是否使用下划线前缀
- 命名是否语义清晰，避免 `data`、`info`、`tmp` 等模糊名称
- 布尔变量是否使用 `is_`、`has_`、`can_` 前缀

### 1.3 类型注解审查
- 公开 API 是否有完整类型注解
- 返回值是否标注（包括 `-> None`）
- 是否使用了 `Any` 可以更精确的类型替代
- 复杂类型是否使用 `TypeAlias` 提高可读性
- `Optional[X]` 是否在不需要 None 的场景被误用

### 1.4 文档字符串审查
- 公开类和方法是否有 docstring
- docstring 风格是否项目内统一（Google / NumPy / Sphinx）
- 是否包含 Args、Returns、Raises 段落
- 示例代码是否可运行（可用 `doctest` 验证）

---

## 二、代码健壮性

### 2.1 异常处理
- 禁止空 `except:` 或 `except Exception:` 不做任何处理
- 是否使用 `raise ... from e` 保留异常链
- 自定义异常是否继承自合理的基类
- 异常消息是否包含足够的上下文信息
- `finally` 块中是否有可能抛出异常覆盖原始异常
- 是否存在过于宽泛的异常捕获

### 2.2 空值与边界
- 函数参数是否处理了 `None` 输入
- 字典访问是否使用 `.get()` 或做 key 存在性检查
- 列表/字符串操作是否处理了空集合
- 数值计算是否处理了除零、溢出
- 切片操作是否考虑了越界

### 2.3 资源管理
- 文件操作是否使用 `with` 语句
- 网络连接是否正确关闭（`httpx.Client` 使用上下文管理器）
- 数据库连接是否使用连接池并正确释放
- 临时文件是否使用 `tempfile` 模块并清理
- 是否存在资源泄漏风险

### 2.4 并发安全
- 多线程场景是否使用 `threading.Lock` 保护共享状态
- 异步代码是否正确使用 `asyncio.Lock`
- 全局可变状态是否线程安全
- `async` 函数中是否误用了阻塞调用（应使用 `run_in_executor`）

---

## 三、设计与架构

### 3.1 模块设计
- 模块职责是否单一，是否存在"上帝模块"
- 循环导入检查
- `__init__.py` 是否只导出公开 API
- 内部实现是否使用 `_` 前缀标记为私有
- 是否存在过度耦合（一个模块修改影响多个模块）

### 3.2 API 设计
- 公开 API 是否稳定，是否有 `@deprecated` 标记
- 函数参数是否过多（超过 5 个应封装为配置对象或 dataclass）
- 是否提供合理的默认值
- 是否遵循最小惊讶原则（方法行为符合名称暗示）
- 返回值类型是否一致（不要同一方法有时返回 list 有时返回 None）

### 3.3 依赖审查
- 运行时依赖是否最小化
- 是否引入了不必要的重量级依赖
- 依赖版本约束是否合理（不要锁死精确版本）
- 是否存在已知安全漏洞的依赖版本

---

## 四、性能审查

### 4.1 常见性能问题
- 循环内是否有重复的 I/O 操作（数据库查询、HTTP 请求）
- 大数据集是否使用生成器（`yield`）而非一次性加载到内存
- 字符串拼接是否使用 `join()` 而非 `+` 循环拼接
- 是否存在不必要的深拷贝（`copy.deepcopy`）
- 列表推导式是否优于等价的 `for` 循环 + `append`

### 4.2 异步性能
- 多个独立的异步调用是否使用 `asyncio.gather()` 并发执行
- 是否存在不必要的 `await`（可以批量处理的场景）
- 连接池大小是否合理配置

### 4.3 缓存策略
- 重复计算是否使用 `functools.lru_cache` 或 `functools.cache`
- 缓存是否有合理的过期策略
- 缓存 key 是否可哈希

---

## 五、安全审查

### 5.1 输入校验
- 外部输入是否做了类型和范围校验
- 是否存在 SQL 注入风险（使用参数化查询）
- 是否存在命令注入风险（`subprocess` 使用列表参数而非字符串）
- 文件路径是否做了路径穿越防护
- 反序列化是否安全（禁止 `pickle.loads` 不可信数据，`yaml.safe_load` 替代 `yaml.load`）

### 5.2 敏感信息
- API Key、密码等是否硬编码在代码中
- 日志中是否打印了敏感信息
- 异常消息是否泄露了内部实现细节
- `.env` 文件是否加入了 `.gitignore`

### 5.3 依赖安全
- 是否使用 `pip-audit` 或 `safety` 检查依赖漏洞
- 是否锁定了依赖版本（`pip freeze` 或 `poetry.lock`）

---

## 六、审查输出格式

对每个发现的问题，按以下格式输出：

```
【问题分类】代码规范 / 健壮性 / 设计架构 / 性能 / 安全
【严重程度】🔴 严重（必须修复）/ 🟡 警告（建议修复）/ 🔵 建议（可优化）
【问题位置】文件名:函数名 (行号)
【问题描述】具体描述问题是什么
【风险说明】不修复可能导致什么后果
【修复建议】具体的修复方案和示例代码
```

审查完成后，给出整体评估：
1. 代码质量评分（1-10 分）
2. 最需要优先修复的 Top 3 问题
3. 架构层面的改进建议（如有）
