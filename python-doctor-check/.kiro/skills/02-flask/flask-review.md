# Flask 应用代码审查 Skill

当用户打开或提交 Flask 项目文件时，按照以下维度进行全面审查。

---

## 一、Flask 专项审查

### 1.1 应用配置
- 是否使用应用工厂模式（`create_app()`），避免全局 `app` 对象
- `SECRET_KEY` 是否硬编码（必须从环境变量读取）
- `DEBUG` 模式是否在生产环境关闭
- 配置是否按环境分离（development / testing / production）
- 数据库 URI 等敏感配置是否通过环境变量注入

### 1.2 路由审查
- URL 设计是否 RESTful（名词复数、HTTP 方法语义正确）
- 是否存在未保护的敏感路由（缺少认证/授权装饰器）
- 路由函数是否过长（超过 30 行应拆分到 Service 层）
- 是否正确使用 HTTP 状态码
- 是否处理了 `request.get_json()` 返回 None 的情况
- URL 参数是否做了类型转换和校验

### 1.3 蓝图审查
- 是否按功能模块划分蓝图
- 蓝图是否设置了 URL 前缀
- 蓝图注册是否在工厂函数中完成
- 是否存在蓝图间的循环导入

### 1.4 上下文与生命周期
- 是否在请求上下文外访问 `request`、`session`、`g` 对象
- 是否正确使用 `app_context()` 和 `request_context()`
- `before_request` / `after_request` 钩子是否合理
- `teardown_appcontext` 是否正确清理资源

---

## 二、数据库审查

### 2.1 SQLAlchemy 使用
- 模型是否有 `created_at` / `updated_at` 时间戳
- 是否存在 N+1 查询问题（缺少 `joinedload` / `subqueryload`）
- 事务是否正确提交/回滚（`db.session.commit()` / `db.session.rollback()`）
- 是否在视图函数中直接写复杂查询（应抽到 Service/Repository 层）
- 批量操作是否使用批量 API 而非循环单条操作
- 是否存在 `SELECT *` 查询（应只查需要的字段）

### 2.2 数据库迁移
- 迁移脚本是否提交到版本控制
- 迁移是否可逆（有 `upgrade` 和 `downgrade`）
- 是否存在破坏性迁移（删列、改类型）未做数据备份

### 2.3 连接管理
- 连接池大小是否合理配置
- 是否存在连接泄漏（未关闭的 session）
- 超时配置是否合理

---

## 三、安全审查

### 3.1 认证与授权
- 登录接口是否有暴力破解防护（限流）
- 密码是否使用 `generate_password_hash` 加密存储
- JWT Token 是否设置了合理的过期时间
- 是否存在越权访问（用户 A 可以访问用户 B 的数据）
- 敏感操作是否有二次验证

### 3.2 注入防护
- SQL 查询是否使用 ORM 参数化，禁止字符串拼接
- 是否存在 SSTI（服务端模板注入）：`render_template_string(user_input)` 是危险的
- 命令执行是否使用 `subprocess` 列表参数
- 文件操作是否防止路径穿越

### 3.3 跨域与 CSRF
- CORS 配置是否合理（不要 `origins="*"`）
- 表单提交是否有 CSRF Token 保护
- Cookie 是否设置了 `HttpOnly`、`Secure`、`SameSite` 属性

### 3.4 响应安全
- 是否设置了安全响应头（`X-Content-Type-Options`、`X-Frame-Options`）
- 错误响应是否泄露了内部信息（堆栈跟踪、SQL 语句）
- API 返回数据是否最小化（不返回密码、内部 ID 等）

---

## 四、性能审查

### 4.1 请求处理
- 是否存在同步阻塞操作（大文件处理、外部 API 调用）
- 是否使用缓存减少重复查询（Flask-Caching）
- 分页查询是否限制了最大页大小
- 文件上传是否限制了大小（`MAX_CONTENT_LENGTH`）

### 4.2 数据库性能
- 查询是否使用了索引
- 是否存在慢查询（复杂 JOIN、子查询）
- 是否存在循环内查询数据库

### 4.3 资源管理
- 大文件是否使用流式处理
- 是否存在内存泄漏（全局变量持续增长）
- 日志是否配置了轮转（避免日志文件过大）

---

## 五、测试审查

### 5.1 测试配置
- 是否使用测试专用配置（内存数据库、禁用 CSRF）
- `conftest.py` 是否提供了 `app`、`client`、`db` fixtures
- 测试是否相互独立（每个测试前后清理数据）

### 5.2 测试覆盖
- API 端点是否有集成测试
- 正常路径和异常路径是否都覆盖
- 认证/授权逻辑是否有测试
- 数据库操作是否有测试

---

## 六、审查输出格式

```
【问题分类】Flask 专项 / 数据库 / 安全 / 性能 / 测试
【严重程度】🔴 严重（必须修复）/ 🟡 警告（建议修复）/ 🔵 建议（可优化）
【问题位置】文件名:函数名 (行号)
【问题描述】具体描述问题是什么
【风险说明】不修复可能导致什么后果
【修复建议】具体的修复方案和示例代码
```

审查完成后，给出整体评估：
1. 代码质量评分（1-10 分）
2. 最需要优先修复的 Top 3 问题
3. 安全风险总结
