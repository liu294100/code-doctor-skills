# Java 代码全面审查 Skill

当用户打开或提交 Java 文件时，请按照以下维度进行全面审查，给出具体问题和改进建议。

---

## 一、代码规范与可读性

### 1.1 命名规范
- 类名：大驼峰（UpperCamelCase），名词或名词短语，如 `OrderService`、`UserRepository`
- 方法名：小驼峰（lowerCamelCase），动词或动词短语，如 `getUserById`、`calculateTotalPrice`
- 常量：全大写下划线分隔，如 `MAX_RETRY_COUNT`、`DEFAULT_TIMEOUT_MS`
- 变量名：小驼峰，语义清晰，禁止单字母变量（循环变量 i/j/k 除外）
- 包名：全小写，使用反向域名，如 `com.company.project.module`
- 布尔变量/方法：使用 `is`、`has`、`can`、`should` 前缀，如 `isValid`、`hasPermission`
- 避免使用拼音命名、无意义缩写（如 `a1`、`tmp2`、`xxxDto`）

### 1.2 代码格式
- 检查缩进是否一致（推荐 4 空格）
- 大括号使用 K&R 风格（左括号不换行）
- 单行代码不超过 120 字符
- 方法体不超过 80 行，超过应拆分
- 类文件不超过 500 行，超过应考虑职责拆分
- 空行使用：方法之间一个空行，逻辑块之间一个空行
- import 语句分组排列：JDK → 第三方库 → 项目内部，禁止使用通配符 `import *`

### 1.3 注释规范
- 类必须有 Javadoc 注释，说明职责和作者
- public 方法必须有 Javadoc，包含 `@param`、`@return`、`@throws`
- 复杂业务逻辑必须有行内注释解释"为什么"而非"做什么"
- 禁止无意义注释，如 `// 获取用户` 对应 `getUser()` 方法
- TODO/FIXME 注释必须包含负责人和日期
- 禁止注释掉的代码残留在主分支

### 1.4 代码整洁度
- 检查是否存在未使用的 import、变量、方法、参数
- 检查是否存在重复代码（DRY 原则）
- 检查魔法数字/字符串是否提取为常量
- 检查是否存在过深的嵌套（超过 3 层应重构）
- 检查方法参数是否过多（超过 4 个应封装为对象）

## 二、代码健壮性与防御性编程

### 2.1 空指针防护
- 所有外部输入（请求参数、RPC 返回、数据库查询结果）必须做空判断
- 使用 `Optional` 替代返回 null
- 集合操作前检查是否为 null 或 empty
- 字符串比较使用 `"constant".equals(variable)` 或 `Objects.equals()`
- 链式调用需注意中间环节可能为 null

### 2.2 异常处理
- 禁止空 catch 块（吞掉异常）
- 禁止 catch `Exception` 或 `Throwable` 的笼统捕获（特殊场景除外需注释说明）
- 异常信息必须包含上下文（关键参数值），便于排查
- 业务异常和系统异常分开处理，使用自定义异常体系
- finally 块中不要有 return 语句
- 使用 try-with-resources 管理可关闭资源
- 异常不要用于控制流程

### 2.3 输入校验
- Controller 层：使用 `@Valid` / `@Validated` + JSR 303 注解校验
- Service 层：对关键业务参数做二次校验（防御性编程）
- 检查数值范围、字符串长度、集合大小的边界
- SQL/NoSQL 查询参数必须参数化，防止注入
- 文件上传检查类型、大小、文件名

### 2.4 边界条件
- 集合操作注意空集合、单元素、大数据量场景
- 数值计算注意溢出（使用 `Math.addExact` 或 BigDecimal）
- 金额计算必须使用 `BigDecimal`，禁止 `float`/`double`
- 日期时间使用 `java.time` API，注意时区处理
- 分页查询限制最大页大小，防止一次查询过多数据

### 2.5 并发安全
- 检查共享可变状态是否有正确的同步机制
- SimpleDateFormat 是线程不安全的，使用 DateTimeFormatter 替代
- HashMap 在并发场景使用 ConcurrentHashMap
- 检查单例 Bean 中是否有可变成员变量（Spring Bean 默认单例）
- 检查 `@Async` 方法的异常处理和线程池配置

## 三、设计原则与模式审查

### 3.1 SOLID 原则
- **单一职责（SRP）**：一个类只负责一个功能领域，Service 不要混合多个业务域逻辑
- **开闭原则（OCP）**：通过抽象和多态扩展功能，而非修改已有代码。检查是否有大量 if-else/switch 可用策略模式替代
- **里氏替换（LSP）**：子类必须能替换父类使用，不要在子类中抛出父类未声明的异常或缩小方法语义
- **接口隔离（ISP）**：接口要小而专，不要定义"胖接口"迫使实现类实现不需要的方法
- **依赖倒置（DIP）**：高层模块依赖抽象而非具体实现，Service 层依赖 Repository 接口而非具体实现类

### 3.2 常见设计模式应用检查
- 大量 if-else 分支 → 建议使用策略模式（Strategy）+ 工厂模式
- 对象创建复杂 → 建议使用建造者模式（Builder），如 Lombok `@Builder`
- 跨系统调用适配 → 建议使用适配器模式（Adapter）
- 事件驱动场景 → 建议使用观察者模式（Observer）/ Spring Event
- 多层装饰增强 → 建议使用装饰器模式（Decorator）
- 状态流转复杂 → 建议使用状态机模式（State Machine）
- 检查是否存在过度设计（Over-Engineering），简单场景不需要复杂模式

### 3.3 分层架构规范
- Controller 层：只做参数接收、校验、结果封装，不包含业务逻辑
- Service 层：核心业务逻辑，事务控制在此层
- Repository/DAO 层：只做数据访问，不包含业务逻辑
- 禁止跨层调用（如 Controller 直接调用 DAO）
- 禁止循环依赖（A → B → A）
- DTO/VO/PO/BO 对象分层使用，不要一个对象贯穿所有层

### 3.4 API 设计规范
- RESTful 风格：资源名用名词复数，HTTP 方法语义正确
- 统一响应格式：`{ code, message, data }` 结构
- 接口版本控制：URL 路径或 Header 方式
- 幂等性设计：POST 创建操作需考虑重复提交
- 接口入参出参不要暴露内部实现细节（如数据库字段名）

## 四、高并发与性能审查

### 4.1 线程池使用
- 禁止使用 `Executors.newFixedThreadPool()` 等快捷方法（可能导致 OOM）
- 必须使用 `ThreadPoolExecutor` 显式指定核心参数：
  - corePoolSize / maximumPoolSize / keepAliveTime
  - workQueue（推荐有界队列 `LinkedBlockingQueue` 并指定容量）
  - threadFactory（自定义线程名便于排查）
  - rejectedExecutionHandler（明确拒绝策略）
- 不同业务使用不同线程池，避免线程池混用导致互相影响
- 检查线程池是否有优雅关闭机制（`shutdown` + `awaitTermination`）

### 4.2 锁与同步
- 优先使用无锁方案：CAS（AtomicXxx）、ThreadLocal、不可变对象
- synchronized 锁粒度要尽量小，避免锁整个方法
- ReentrantLock 使用时必须在 finally 中 unlock
- 读多写少场景使用 `ReadWriteLock` 或 `StampedLock`
- 分布式环境使用分布式锁（Redis/Zookeeper），注意：
  - 锁超时时间设置合理
  - 加锁和解锁必须是同一个持有者
  - 考虑锁续期机制（如 Redisson 看门狗）
- 检查是否存在死锁风险（多把锁的获取顺序不一致）

### 4.3 缓存设计
- 缓存穿透防护：布隆过滤器 / 缓存空值（设置短过期时间）
- 缓存击穿防护：热点 key 使用互斥锁或永不过期 + 异步刷新
- 缓存雪崩防护：过期时间加随机值，避免同时失效
- 缓存与数据库一致性：推荐 Cache-Aside 模式（先更新 DB，再删缓存）
- 大 Value 拆分，避免 Redis 大 Key（String < 10KB，集合元素 < 5000）
- 缓存预热策略：系统启动时加载热点数据
- 多级缓存架构：本地缓存（Caffeine）→ 分布式缓存（Redis）→ DB

### 4.4 数据库优化
- 索引检查：WHERE/JOIN/ORDER BY 字段是否有索引
- 避免全表扫描：禁止 `SELECT *`，禁止无 WHERE 条件的查询
- 大表查询必须分页，且禁止深分页（使用游标分页或 seek 方式）
- 批量操作使用 `batch insert/update`，避免循环单条操作
- 事务范围尽量小，不要在事务中做 RPC 调用或耗时操作
- 读写分离场景注意主从延迟问题
- 分库分表场景注意分片键选择和跨分片查询

### 4.5 异步与消息队列
- 异步操作必须有超时控制和异常处理
- CompletableFuture 链式调用注意异常传播
- MQ 消费者必须保证幂等性（通过唯一 ID 去重）
- 消息发送考虑可靠性：本地消息表 / 事务消息
- 消费失败的重试策略和死信队列处理
- 消息顺序性要求时使用顺序消息或分区有序

## 五、高可用与容错设计

### 5.1 服务容错
- 外部调用（HTTP/RPC/DB）必须设置超时时间，禁止无限等待
- 关键服务调用实现熔断机制（Sentinel / Resilience4j）
  - 熔断阈值、恢复时间、半开状态配置是否合理
- 服务降级策略：核心链路保护，非核心功能可降级
- 重试机制：
  - 只对可重试的异常重试（网络超时、5xx），不要重试业务异常（4xx）
  - 重试次数有限（通常 2-3 次）
  - 使用指数退避（Exponential Backoff）+ 随机抖动（Jitter）
- 限流策略：
  - 接口级别限流（QPS 限制）
  - 用户级别限流（防刷）
  - 滑动窗口 / 令牌桶 / 漏桶算法选择

### 5.2 分布式事务
- 优先使用最终一致性方案，避免强一致性（性能代价大）
- 常见方案评估：
  - 本地消息表 + MQ：适合异步场景
  - TCC（Try-Confirm-Cancel）：适合强一致性要求
  - Saga 模式：适合长事务链路
  - Seata AT 模式：适合简单场景快速接入
- 补偿机制：每个正向操作都要有对应的补偿/回滚操作
- 对账机制：定时任务核对数据一致性

### 5.3 幂等性设计
- 所有写操作接口必须保证幂等
- 常见方案：
  - 唯一索引 / 唯一约束
  - Token 机制（请求前获取 token，请求时携带并校验）
  - 状态机控制（只允许合法的状态流转）
  - 乐观锁（version 字段）
  - 分布式锁 + 业务校验

### 5.4 监控与可观测性
- 关键业务操作是否有日志记录（入参、出参、耗时、异常）
- 日志级别使用是否合理（ERROR/WARN/INFO/DEBUG）
- 日志中是否包含 traceId 便于链路追踪
- 是否有关键指标埋点（QPS、RT、成功率、业务指标）
- 慢查询、慢接口是否有告警机制
- 禁止在日志中打印敏感信息（密码、token、身份证号等）

## 六、安全审查

### 6.1 认证与授权
- 接口是否有权限控制（`@PreAuthorize`、自定义注解拦截）
- 敏感操作是否有二次验证
- Token/Session 过期时间是否合理
- 密码存储必须加密（BCrypt），禁止明文或 MD5

### 6.2 数据安全
- SQL 参数化查询，防止 SQL 注入
- XSS 防护：输出编码、输入过滤
- 敏感数据（手机号、身份证、银行卡）脱敏展示
- 接口返回数据最小化原则，不返回不必要的字段
- 文件上传路径不可由用户控制，防止路径穿越
- 序列化/反序列化安全：禁用不安全的反序列化（如 Java 原生序列化）

### 6.3 接口安全
- 防重放攻击：timestamp + nonce + sign
- 接口限流防刷
- 关键操作审计日志
- CORS 配置是否合理（不要使用 `*`）

## 七、Spring Boot / Spring Cloud 专项

### 7.1 Bean 管理
- 检查循环依赖（`@Lazy` 只是临时方案，应从设计上消除）
- `@Autowired` 推荐使用构造器注入而非字段注入
- `@Value` 配置项是否有默认值，防止启动失败
- `@Transactional` 注意事项：
  - 只对 public 方法生效
  - 同类内部调用不会触发代理（事务失效）
  - rollbackFor 默认只回滚 RuntimeException，需显式指定 `rollbackFor = Exception.class`
  - 避免大事务，事务中不要有 RPC/HTTP 调用

### 7.2 配置管理
- 敏感配置（数据库密码、密钥）不要硬编码，使用配置中心或环境变量
- 不同环境（dev/test/prod）配置隔离
- 连接池参数是否合理（数据库连接池、HTTP 连接池、Redis 连接池）

### 7.3 微服务设计
- 服务拆分粒度是否合理（不要过细导致调用链过长）
- 服务间调用是否有降级兜底
- 是否存在分布式事务问题未处理
- 配置中心、注册中心的高可用
- 网关层统一鉴权、限流、日志

## 八、审查输出格式

对每个发现的问题，请按以下格式输出：

```
【问题分类】代码规范 / 健壮性 / 设计原则 / 高并发 / 高可用 / 安全 / Spring 专项
【严重程度】🔴 严重（必须修复）/ 🟡 警告（建议修复）/ 🔵 建议（可优化）
【问题位置】类名#方法名 (行号)
【问题描述】具体描述问题是什么
【风险说明】不修复可能导致什么后果
【修复建议】具体的修复方案和示例代码
```

审查完成后，给出整体评估：
1. 代码质量评分（1-10 分）
2. 最需要优先修复的 Top 3 问题
3. 架构层面的改进建议（如有）
