# 互联网券商架构设计与审查 Skill

针对互联网券商（股票、期货、期权、基金等）场景的架构设计审查。
涵盖行情系统、交易系统、清结算、账户体系、风控、合规等券商核心领域。

## 二、行情系统架构

### 2.1 行情数据流

```
交易所/信息商 → 行情接入网关（协议解析）
             → 行情处理引擎（清洗/聚合/指标计算）
             → 行情分发层
               ├→ 推送服务（WebSocket → 客户端）
               ├→ 缓存层（Redis → API 查询）
               ├→ 存储层（时序DB → 历史行情）
               └→ 内部消费（风控/交易/策略）
```

### 2.2 行情系统设计要点

**延迟要求**
- Level 1 行情（快照）：交易所到客户端 < 200ms
- Level 2 行情（逐笔）：交易所到客户端 < 100ms
- 内部行情（风控/交易引用）：< 10ms
- K 线聚合延迟：< 1s

**行情接入层**
- 多源接入：交易所直连 + 信息商备份，自动切换
- 协议适配：不同交易所协议不同（STEP/Binary/FIX），统一转换为内部格式
- 行情去重：同一笔行情可能从多个源收到，按序列号去重
- 断线重连：自动重连 + 数据补全（请求缺失的序列号段）
- 时钟同步：NTP 精确到毫秒级，行情时间戳以交易所时间为准

**行情处理引擎**
- 快照合成：逐笔数据合成 Level 1 快照（最新价/涨跌幅/成交量等）
- K 线聚合：实时聚合 1min/5min/15min/30min/60min/日/周/月 K 线
  - 使用时间窗口聚合，注意跨交易日/跨周/跨月的边界处理
  - 复权计算：前复权/后复权，除权除息日重新计算历史 K 线
- 指标计算：MA/EMA/MACD/RSI/BOLL 等技术指标实时计算
- 涨跌停价计算：根据前收盘价和涨跌幅限制实时计算

**行情推送架构**
- WebSocket 长连接：每个连接订阅不同的股票
- 推送频率控制：
  - 活跃股票：每 200ms 推送一次快照
  - 非活跃股票：有变化才推送（变化推送）
  - 客户端不可见时降低推送频率
- 推送压缩：增量推送（只推送变化的字段），Protobuf 序列化
- 连接管理：
  - 单机支撑 10 万+ WebSocket 连接（Netty + Epoll）
  - 按股票分组广播，避免逐连接推送
  - 心跳检测 + 自动断开僵尸连接

**行情存储**
- 实时行情：Redis（Hash 结构，field 为股票代码）
- Tick 数据：时序数据库（InfluxDB/TDengine/QuestDB）
- 历史 K 线：ClickHouse（列式存储，聚合查询快）
- 财务数据：MySQL/PostgreSQL（结构化，更新频率低）

**CAP 选型：AP**
- 行情允许短暂延迟和不一致，优先保证推送可用性
- 多级缓存：本地缓存（最新快照）→ Redis → 时序 DB
- 降级策略：实时行情不可用时展示最近快照 + 提示延迟

### 2.3 行情性能优化

- 内存映射：行情快照使用 Off-Heap 内存，避免 GC
- 批量推送：攒批 50ms 内的变化，一次性推送（减少网络包数量）
- 零拷贝：Netty FileRegion / CompositeByteBuf
- 行情压缩：相邻快照 Delta 编码，压缩率可达 70%+
- 多播（Multicast）：内部行情分发使用 UDP 多播（同一数据只发一次）

## 三、交易系统架构

### 3.1 委托交易链路

```
客户端下单 → API Gateway（鉴权/限流）
           → 交易前置（参数校验/证券信息校验）
           → 风控引擎（事前风控，同步 < 5ms）
           → 委托处理（生成委托单/冻结资金和股份）
           → 报盘网关（转换为交易所协议/发送到交易所）
           → 交易所撮合
           → 回报处理（成交回报/委托确认）
           → 资产更新（持仓/资金/盈亏）
           → 客户端推送（WebSocket 推送成交通知）
```

### 3.2 交易系统设计要点

**委托管理**
- 委托状态机：
  ```
  已报 → 已确认 → 部分成交 → 全部成交
    ↓       ↓         ↓
  废单   已撤(待撤→已撤)  部撤
  ```
- 委托类型支持：限价单、市价单、条件单、止损单、止盈单、冰山单、暗盘单
- 条件单引擎：
  - 监听行情触发条件（价格触发/指标触发/时间触发）
  - 条件满足后自动转为普通委托提交
  - 注意：条件判断必须用行情快照，不能用缓存（可能延迟）
- 委托校验：
  - 交易时间校验（开市/收市/集合竞价/盘后交易）
  - 涨跌停价格校验
  - 最小/最大委托数量校验（A 股 100 股整数倍，科创板/创业板可 1 股）
  - 证券停牌/退市校验
  - 账户交易权限校验（创业板/科创板/港股通/融资融券需单独开通）

**报盘网关**
- 与交易所的通信协议：
  - 上交所：STEP 协议（基于 FIX 扩展）
  - 深交所：Binary 协议
  - 港交所：OCG 协议（基于 FIX）
  - 美股：FIX 4.2/4.4 协议
- 报盘队列：
  - 按交易所分队列，独立发送
  - 流控：遵守交易所的报盘速率限制（如上交所每秒限制笔数）
  - 序号管理：每笔报盘有唯一序号，断线重连后按序号补发
- 回报处理：
  - 异步接收交易所回报
  - 回报去重（同一笔回报可能收到多次）
  - 回报顺序保证（按序号排序处理）
  - 回报与委托匹配（通过委托编号关联）

**CAP 选型：CP**
- 交易必须强一致，不能出现重复下单或丢单
- 委托状态流转必须严格按状态机执行
- 资金/股份冻结与委托提交在同一事务中

### 3.3 交易性能优化

**目标指标**
- 委托提交到交易所：< 5ms（内部处理）
- 成交回报到客户端：< 50ms
- 系统吞吐量：> 10000 笔/秒（单市场）
- 开盘集合竞价峰值：正常流量的 5-10 倍

**关键优化**
- 委托处理单线程化：按账户分片，每个分片单线程处理，消除锁竞争
- 资金/持仓内存化：热点账户数据加载到内存，减少 DB 访问
- 报盘批量发送：攒批 1ms 内的委托，批量发送到交易所
- 回报异步处理：成交回报写入 Disruptor 队列，异步更新持仓和资金
- 预计算：开盘前预加载证券信息、涨跌停价、账户权限等静态数据

### 3.4 多市场交易支持

| 市场 | 交易时间 | 最小单位 | 涨跌幅 | T+N | 货币 |
|------|---------|---------|--------|-----|------|
| A 股（沪/深） | 9:30-15:00 | 100 股（科创板 1 股） | ±10%（科创/创业 ±20%） | T+1 | CNY |
| 港股 | 9:30-16:00 | 1 手（每股不同） | 无涨跌幅限制 | T+2 | HKD |
| 美股 | 9:30-16:00 ET | 1 股 | 熔断机制 | T+1 | USD |
| 期货 | 9:00-15:00 + 夜盘 | 1 手 | 品种不同 | T+0 | CNY |
| 期权 | 9:30-15:00 | 1 张 | 品种不同 | T+0 | CNY |

**多市场设计要点**
- 交易日历：每个市场独立的交易日历（节假日不同）
- 汇率处理：多币种账户，实时汇率换算，汇率锁定机制
- 时区处理：统一使用 UTC 存储，展示时转换为当地时区
- 交易规则引擎：每个市场的交易规则抽象为配置，不硬编码

## 四、账户与资产体系

### 4.1 账户体系设计

```
客户 (1) ──→ (N) 证券账户（沪A/深A/港股/美股）
  │
  └──→ (N) 资金账户（人民币/港币/美元）
  │
  └──→ (1) 信用账户（融资融券，可选）
```

**证券账户**
- 一个客户可开多个市场的证券账户
- 证券账户由中登公司/交易所统一编号
- 账户状态：正常、冻结、休眠、注销
- 账户权限：按市场/品种独立控制（科创板需单独开通）

**资金账户**
- 多币种独立账户，每个币种独立余额
- 资金字段：总资产、可用余额、冻结金额、在途资金
- 资金变动类型：买入冻结、卖出解冻、银证转入、银证转出、分红入账、利息、手续费
- 资金精度：人民币精确到分（2 位小数），美元精确到 cent

**持仓管理**
- 持仓字段：证券代码、持仓数量、可用数量、冻结数量、成本价、市值、盈亏
- 成本价计算：
  - 买入均价法（默认）：总成本 / 总数量
  - 移动加权平均法
  - 注意：部分卖出后成本价重新计算
- T+1 处理（A 股）：
  - 今日买入的股份，今日不可卖出
  - 数据结构：`可用数量 = 总数量 - 冻结数量 - 今日买入数量`
  - 日终清算时：今日买入数量归零，变为可用

### 4.2 资产计算

**实时市值计算**
```
总资产 = 资金余额 + Σ(持仓数量 × 最新价) + 在途资金
浮动盈亏 = Σ((最新价 - 成本价) × 持仓数量)
日盈亏 = Σ((最新价 - 昨收价) × 持仓数量) + 今日已实现盈亏
收益率 = 浮动盈亏 / 投入成本 × 100%
```

**性能优化**
- 市值计算不能每次都遍历所有持仓 × 查行情
- 方案：行情变化时，增量更新受影响的持仓市值
- 使用 Redis Sorted Set 存储持仓市值，行情变化时批量更新
- 客户端拉取时直接读缓存，不实时计算

## 五、清结算系统

### 5.1 日终清算流程

```
收市 → 数据准备（接收交易所清算文件）
     → 数据核对（本地成交 vs 交易所成交，必须 100% 一致）
     → 资金清算（计算每个账户的资金变动）
       ├→ 成交资金清算（买入扣款/卖出入账）
       ├→ 手续费清算（佣金/印花税/过户费/规费）
       ├→ 利息清算（融资利息/闲置资金利息）
       └→ 公司行为处理（分红/送股/配股/拆合股）
     → 股份清算（更新持仓数量，T+1 可用处理）
     → 资金交收（与中登/银行的资金划拨，T+1/T+2）
     → 报表生成（客户对账单/监管报表）
     → 清算完成 → 初始化下一交易日
```

### 5.2 清算设计要点

**数据核对（最关键环节）**
- 逐笔核对：本地成交记录 vs 交易所清算文件
- 核对维度：成交数量、成交价格、成交金额、手续费
- 差异处理：
  - 自动调整：金额差异 < 1 分（四舍五入差异）
  - 人工介入：其他差异必须人工确认后处理
  - 差异未处理完毕，清算不能继续

**手续费计算**
```
A 股手续费 = 佣金 + 印花税 + 过户费 + 规费
佣金 = max(成交金额 × 佣金费率, 最低佣金)  // 通常最低 5 元
印花税 = 成交金额 × 0.05%（仅卖出收取）
过户费 = 成交金额 × 0.001%
规费 = 经手费 + 证管费
```
- 费率配置化：不同客户可能有不同佣金费率
- 费率精度：费率精确到小数点后 6 位
- 计算结果：四舍五入到分

**公司行为处理（Corporate Action）**
- 现金分红：除权日自动入账，注意税率（持股时间不同税率不同）
- 送股/转增：除权日自动增加持仓，调整成本价
- 配股：生成配股权证，客户选择是否行权
- 拆合股：调整持仓数量和成本价
- 要约收购/退市：特殊处理流程

**CAP 选型：CP**
- 清算必须强一致，任何差异都可能导致资金错误
- 清算过程中系统不对外提供交易服务（维护窗口）
- 清算失败必须支持断点续跑和全量重跑

### 5.3 清算性能

- 目标：100 万客户的日终清算 < 30 分钟
- 并行清算：按客户分片并行处理，无依赖的步骤并行执行
- 批量 DB 操作：batch insert/update，每批 1000-5000 条
- 清算前预加载：证券信息、费率配置、汇率等加载到内存

## 六、券商风控体系

### 6.1 三层风控架构

**事前风控（交易前，同步，< 5ms）**
```
委托请求 → 风控引擎
          ├→ 账户级校验
          │   ├→ 账户状态（是否正常/冻结/限制交易）
          │   ├→ 交易权限（是否开通该市场/品种权限）
          │   └→ 适当性检查（风险等级是否匹配产品风险）
          ├→ 资金/持仓校验
          │   ├→ 买入：可用资金 >= 委托金额 + 预估手续费
          │   ├→ 卖出：可用股份 >= 委托数量
          │   └→ 集中度检查（单只股票持仓不超过总资产 X%）
          ├→ 交易规则校验
          │   ├→ 价格校验（涨跌停范围内）
          │   ├→ 数量校验（最小单位/最大数量）
          │   └→ 频率校验（防止高频刷单）
          └→ 合规校验
              ├→ 内幕交易监控（限制名单/窗口期）
              ├→ 关联交易检查
              └→ 反洗钱规则
```

**事中风控（交易中，准实时）**
- 持仓集中度监控：单客户/单产品持仓比例
- 异常交易监控：短时间内大量撤单、对倒交易、尾盘拉升
- 保证金监控（融资融券）：维持担保比例实时计算，低于警戒线自动通知
- 强制平仓触发：维持担保比例低于平仓线，自动生成平仓委托

**事后风控（交易后，T+0/T+1）**
- 异常交易分析报告
- 客户交易行为画像
- 监管报送数据准备
- 风控规则效果评估和调优

### 6.2 风控引擎性能

- 规则数量：通常 100-500 条规则
- 执行方式：规则预编译为决策树，内存执行
- 数据依赖：账户信息/持仓/行情全部内存化
- 目标延迟：单笔风控 < 5ms（P99）
- 规则热更新：新规则加载不影响在线交易

### 6.3 融资融券风控（两融业务专项）

**维持担保比例计算**
```
维持担保比例 = (现金 + 信用证券市值 × 折算率) / (融资负债 + 融券负债市值)
```
- 警戒线：通常 140%（通知客户追加担保物）
- 平仓线：通常 130%（强制平仓）
- 实时计算：行情变化时重新计算所有两融客户的担保比例
- 批量通知：低于警戒线自动发送短信/App 推送/电话通知

## 七、开户与 KYC

### 7.1 在线开户流程

```
身份验证 → OCR 识别（身份证正反面）
         → 活体检测（人脸识别 + 动作验证）
         → 人脸比对（与公安库/身份证照片比对）
         → 风险测评（适当性问卷）
         → 协议签署（电子签名）
         → 账户开立（调用中登公司接口）
         → 银行三方存管签约
         → 开户完成
```

### 7.2 KYC 与适当性管理

**客户风险等级**
- 保守型（C1）→ 稳健型（C2）→ 平衡型（C3）→ 积极型（C4）→ 激进型（C5）

**产品风险等级**
- 低风险（R1）：国债、货币基金
- 中低风险（R2）：债券基金、银行理财
- 中风险（R3）：混合基金、股票
- 中高风险（R4）：期货、期权、融资融券
- 高风险（R5）：场外衍生品、分级基金 B

**适当性匹配规则**
- 客户风险等级 >= 产品风险等级 → 允许交易
- 客户风险等级 < 产品风险等级 → 风险提示 + 客户确认（或直接禁止）
- 特殊品种（科创板/创业板/北交所）：额外的资产和交易经验要求

## 八、多市场特殊处理

### 8.1 港股通专项

**汇率处理**
- 参考汇率：每日开盘前由中国结算公布
- 结算汇率：T+2 日由中国结算公布
- 买入时：按参考汇率冻结资金（上浮 3% 作为汇率缓冲）
- 结算时：按结算汇率实际扣款，多退少补
- 汇率差异可能导致资金不足，需要风控预留

**额度控制**
- 每日额度：港股通每日额度限制（目前 520 亿）
- 额度用完后不能再买入，但可以卖出
- 额度实时查询：对接交易所额度接口

**交易差异**
- 最小交易单位：每手股数不同（100/200/500/1000/2000 等）
- 无涨跌幅限制（但有市场波动调节机制 VCM）
- 碎股处理：不足一手的股份只能卖出不能买入
- 暗盘交易：IPO 暗盘交易支持

### 8.2 美股专项

**交易时段**
- 盘前交易：4:00-9:30 ET
- 正常交易：9:30-16:00 ET
- 盘后交易：16:00-20:00 ET
- 注意夏令时/冬令时切换

**特殊规则**
- PDT 规则（Pattern Day Trader）：账户资产 < $25,000 时，5 个交易日内日内交易不超过 3 次
- 碎股交易：支持 0.001 股（部分券商）
- 期权交易：美式期权可提前行权
- ADR 费用：持有 ADR 需收取托管费

### 8.3 期货/期权专项

**保证金制度**
```
持仓保证金 = 合约价值 × 保证金比例
可用资金 = 总权益 - 持仓保证金 - 冻结保证金
风险度 = 持仓保证金 / 总权益 × 100%
```
- 保证金比例：交易所保证金 + 期货公司加收
- 逐日盯市（Mark to Market）：每日结算盈亏，保证金不足需追加
- 强平线：风险度 > 100% 触发强制平仓

**夜盘处理**
- 夜盘属于下一交易日
- 夜盘和日盘的持仓/资金连续计算
- 系统设计需要处理跨自然日的交易日概念

## 九、券商系统高可用设计

### 9.1 交易时段可用性要求

| 时段 | 可用性要求 | 允许停机 | 说明 |
|------|-----------|---------|------|
| 集合竞价（9:15-9:30） | 99.99% | < 5s/天 | 开盘关键期 |
| 连续竞价（9:30-15:00） | 99.999% | < 0.3s/天 | 核心交易时段 |
| 盘后（15:00-17:00） | 99.9% | < 5min/天 | 盘后交易/查询 |
| 清算时段（17:00-次日8:00） | 99% | 可维护 | 日终清算 |

### 9.2 关键容灾设计

**交易系统容灾**
- 主备热切换：交易引擎主备部署，主节点故障 < 3s 切换到备节点
- 委托恢复：备节点启动后从 WAL 日志恢复未完成委托
- 报盘网关：双通道连接交易所，主通道故障自动切换备通道
- 回报补全：切换后主动向交易所查询缺失的回报

**行情系统容灾**
- 多源热备：同时连接多个行情源，主源故障自动切换
- 本地缓存：行情推送服务本地缓存最新快照，上游故障时仍可提供最近数据
- 降级展示：行情延迟超过阈值，客户端展示"行情延迟"提示

**数据库容灾**
- 资金库：同城双机房主从半同步，RPO = 0
- 交易库：同城双机房，异步复制，RPO < 1s
- 行情库：多副本，允许丢失最近几秒数据

### 9.3 开盘前检查清单

每个交易日开盘前必须完成：
- [ ] 日终清算是否正常完成
- [ ] 证券信息是否更新（新股上市/停复牌/涨跌停价）
- [ ] 交易日历是否正确（是否交易日）
- [ ] 交易所连接是否正常（报盘通道/回报通道/行情通道）
- [ ] 银行通道是否正常（银证转账）
- [ ] 风控规则是否加载完成
- [ ] 系统时钟是否同步（NTP）
- [ ] 各服务健康检查是否通过
- [ ] 监控告警是否正常工作

## 十、监管合规

### 10.1 监管报送

**日常报送**
- 客户交易数据报送（每日）
- 融资融券业务数据报送（每日）
- 客户资金存管数据报送（每日）
- 反洗钱大额/可疑交易报告

**定期报送**
- 净资本计算表（月度）
- 风险控制指标（月度）
- 客户投诉处理报告（季度）
- 信息安全评估报告（年度）

### 10.2 合规检查项

- [ ] 客户资金第三方存管（客户资金在银行，券商不能挪用）
- [ ] 适当性管理（客户风险等级与产品匹配）
- [ ] 信息隔离墙（研究/投行/自营之间信息隔离）
- [ ] 内幕交易防控（限制名单/监控名单管理）
- [ ] 反洗钱（客户身份识别/交易监控/可疑报告）
- [ ] 投资者保护（风险提示/信息披露/投诉处理）
- [ ] 数据安全（客户数据加密/访问控制/审计日志）
- [ ] 业务连续性（灾备方案/应急预案/定期演练）

## 十二、架构审查输出格式

对每个发现的问题，请按以下格式输出：

```
【业务域】行情 / 交易 / 清算 / 账户 / 风控 / 合规 / 基础设施
【风险等级】🔴 P0 交易事故（可能导致客户资金损失或交易中断）
           🟠 P1 业务风险（可能导致数据不一致或合规问题）
           🟡 P2 性能风险（可能导致延迟增大或容量不足）
           🔵 P3 优化建议（可改善但不紧急）
【问题描述】具体描述问题
【影响评估】影响哪些业务场景、多少客户、潜在损失
【根因分析】技术层面的根本原因
【修复方案】具体修复步骤和代码示例
【验证方式】如何验证修复效果
【合规影响】是否涉及监管合规要求
```

审查完成后给出：
1. 交易安全评分（1-10）
2. 系统可用性评分（1-10）
3. 性能达标评估（各子系统延迟/吞吐是否达标）
4. 合规风险评估
5. 最需要优先处理的 Top 3 风险项
6. 架构演进建议
