# 互联网金融架构设计与审查 Skill

针对互联网金融场景的架构设计审查，涵盖 CAP 选型策略、极致性能优化、金融级可靠性设计。
审查代码或架构方案时，请按以下维度进行全面评估。

---

## 一、CAP 定理在金融场景的选型策略

### 核心原则
金融系统不是简单的"选 CP 还是 AP"，而是按业务域精细化选型。同一个系统内不同模块可以有不同的 CAP 倾向。

### 1.1 必须选 CP（强一致性）的场景

以下场景数据不一致会直接导致资金损失或合规风险，必须牺牲部分可用性保证一致性：

**资金账户核心**
- 用户余额变更（充值、提现、转账、扣款）
- 账户开户、销户操作
- 资金冻结、解冻操作
- 技术方案：同步写入 + 分布式事务（TCC/Saga），数据库主库强一致读
- 数据库选型：MySQL InnoDB（ACID），主从半同步复制（semi-sync replication）
- 禁止最终一致性：余额不能出现"暂时多扣"或"暂时多加"

**交易撮合引擎**
- 订单匹配、成交确认
- 价格优先、时间优先的排序必须严格一致
- 技术方案：单线程撮合（如 LMAX Disruptor 模式），内存撮合 + WAL 持久化
- 集群方案：主备热切换（非多主），同一时刻只有一个撮合节点处理

**清结算系统**
- 日终清算、资金划拨
- 对账文件生成与核对
- 技术方案：批处理 + 强事务，失败必须全部回滚
- 对账必须 100% 一致，差异必须人工介入

**风控决策（实时）**
- 交易风控拦截（反欺诈、反洗钱）
- 额度控制（单笔限额、日累计限额）
- 技术方案：同步调用，宁可拒绝交易也不能放过风险
- 风控规则引擎必须强一致加载，不能出现新旧规则并存

**合规与审计**
- 交易流水记录（不可篡改、不可丢失）
- 监管报送数据
- 技术方案：WAL + 同步落盘，流水表只追加不修改不删除
- 审计日志独立存储，与业务库物理隔离

### 1.2 可以选 AP（高可用优先）的场景

以下场景短暂不一致可接受，优先保证服务可用和用户体验：

**行情展示系统**
- 股票/基金/数字货币实时行情
- K 线图、深度图展示
- 允许短暂延迟（秒级），不同用户看到的价格可以有微小差异
- 技术方案：Redis Cluster（AP）+ 本地缓存，多级缓存架构
- 降级策略：缓存不可用时展示最近一次快照

**用户中心与营销**
- 用户基本信息（昵称、头像、偏好设置）
- 积分、优惠券、活动页面
- 消息通知、站内信
- 技术方案：最终一致性，异步同步，读写分离
- 积分变更可以异步处理，短暂不一致用户无感知

**内容与资讯**
- 财经新闻、研报、公告
- 社区帖子、评论
- 技术方案：CDN + 缓存，最终一致性

**日志与监控数据**
- 操作日志查询展示
- 监控指标聚合展示
- 技术方案：Elasticsearch（AP），允许短暂查询不到最新数据

### 1.3 需要精细权衡的场景

**支付网关**
- 支付发起：AP（保证用户能提交支付请求）
- 支付确认：CP（最终扣款结果必须一致）
- 方案：异步确认模式，先受理（AP）再确认（CP）
- 支付状态查询：允许短暂"处理中"，但最终状态必须准确

**订单系统**
- 下单：AP（高峰期优先保证能下单）
- 订单状态流转：CP（状态机严格一致）
- 方案：下单写入消息队列（AP），后续处理强一致

**对账系统**
- 实时对账：AP（允许短暂差异，定时补偿）
- 日终对账：CP（必须 100% 一致）

## 二、极致性能优化策略

### 2.1 交易链路性能（目标：核心交易 < 50ms，行情推送 < 10ms）

**内存计算优先**
- 热点数据全部加载到内存：用户余额、持仓、风控规则
- 使用堆外内存（Off-Heap）减少 GC 影响：ByteBuffer.allocateDirect / Unsafe
- 内存数据结构选型：
  - 高频读写：ConcurrentHashMap（分段锁）或 Caffeine Cache
  - 有序集合：ConcurrentSkipListMap（无锁读）
  - 计数器：LongAdder（优于 AtomicLong，减少 CAS 竞争）
  - 位图运算：BitSet / RoaringBitmap（用户标签、布隆过滤器）

**无锁化设计**
- 撮合引擎：单线程 + Disruptor 无锁队列
  - Ring Buffer 预分配，零 GC
  - 事件驱动，批量处理
  - 单线程避免锁竞争，吞吐量可达百万级/秒
- 账务处理：分账户分片，每个分片单线程处理，消除锁竞争
- 行情推送：使用 Mechanical Sympathy 理念，CPU 缓存行对齐，避免伪共享（False Sharing）
  - `@Contended` 注解或手动 padding

**网络层优化**
- 使用 Netty 自定义协议替代 HTTP（减少协议开销）
- 核心链路使用 TCP 长连接 + 连接池
- 序列化选型：
  - 内部通信：Protobuf / FlatBuffers（零拷贝反序列化）
  - 对外接口：JSON（兼容性）
- 启用 TCP_NODELAY 禁用 Nagle 算法（降低延迟）
- SO_REUSEPORT 多线程监听同一端口（Linux 内核级负载均衡）

**数据库极致优化**
- 写入优化：
  - 批量写入（batch size 500-1000）
  - 异步刷盘（innodb_flush_log_at_trx_commit = 2，非资金场景）
  - 资金场景必须 innodb_flush_log_at_trx_commit = 1（每次事务刷盘）
- 读取优化：
  - 覆盖索引（Index Only Scan），避免回表
  - 查询结果集控制在 1000 条以内
  - 预编译 SQL（PreparedStatement 缓存）
- 连接池调优：
  - HikariCP（最快的连接池）
  - minimumIdle = maximumPoolSize（预热连接，避免冷启动）
  - connectionTimeout < 3s，idleTimeout = 10min
- 分库分表策略：
  - 用户维度：userId 取模分库
  - 时间维度：交易流水按月分表
  - 全局唯一 ID：Snowflake 算法（时间戳 + 机器ID + 序列号）

### 2.2 JVM 调优（金融场景专项）

**GC 选型**
- 低延迟场景（撮合引擎、风控）：ZGC 或 Shenandoah
  - 目标：GC 停顿 < 1ms
  - `-XX:+UseZGC -XX:ZCollectionInterval=0 -XX:ZAllocationSpikeTolerance=5`
- 高吞吐场景（批处理、清算）：G1GC
  - `-XX:+UseG1GC -XX:MaxGCPauseMillis=50 -XX:G1HeapRegionSize=16m`
- 通用服务：G1GC 即可满足

**内存配置**
- 堆内存：物理内存的 50-70%，预留给堆外和 OS 缓存
- 新生代：堆的 30-40%（金融场景对象生命周期短）
- 元空间：`-XX:MetaspaceSize=256m -XX:MaxMetaspaceSize=512m`
- 直接内存：`-XX:MaxDirectMemorySize=2g`（Netty 堆外内存）
- 开启大页：`-XX:+UseLargePages`（减少 TLB miss）

**JIT 编译优化**
- `-XX:+TieredCompilation`（分层编译，默认开启）
- `-XX:CompileThreshold=5000`（热点方法编译阈值）
- 关键路径代码预热：启动时模拟请求触发 JIT 编译
- 考虑 GraalVM Native Image 用于冷启动敏感的 Serverless 场景

### 2.3 缓存架构（多级缓存极致方案）

```
请求 → L1 本地缓存（Caffeine, < 1ms）
     → L2 分布式缓存（Redis Cluster, < 5ms）
     → L3 数据库（MySQL/TiDB, < 50ms）
```

**L1 本地缓存（Caffeine）**
- 行情数据：expireAfterWrite = 1s，maximumSize = 10万
- 风控规则：expireAfterWrite = 30s，refreshAfterWrite = 10s（异步刷新）
- 用户基本信息：expireAfterWrite = 5min
- 注意：本地缓存更新需要通过 Redis Pub/Sub 或 MQ 广播通知所有节点

**L2 Redis 缓存**
- 集群模式：Redis Cluster（至少 6 节点，3 主 3 从）
- 热点 Key 处理：
  - 本地缓存兜底（L1 命中则不访问 Redis）
  - Redis 热点 Key 探测 + 本地缓存自动升级
- Pipeline 批量操作：减少网络往返
- Lua 脚本：原子操作（如库存扣减、限流计数）

**缓存一致性方案（金融级）**
- 资金相关：不使用缓存，直接读主库
- 非资金但一致性要求高：Cache-Aside + 延迟双删
  1. 更新 DB
  2. 删除缓存
  3. 延迟 500ms 再删一次（防止并发读写导致脏数据）
- 非资金低一致性：异步更新缓存，允许秒级延迟

### 2.4 异步化与削峰

**消息队列选型**
- 交易消息（可靠性优先）：RocketMQ（事务消息、顺序消息）
- 行情推送（吞吐优先）：Kafka（百万级 TPS）
- 实时通知（低延迟）：Redis Stream 或 Pulsar

**削峰填谷方案**
```
用户请求 → 网关限流（令牌桶）
         → 请求写入 MQ（削峰）
         → 消费者按固定速率处理（填谷）
         → 处理结果异步通知用户
```
- 秒杀/抢购场景：
  - 前端：按钮防抖 + 验证码 + 倒计时
  - 网关：用户级限流（同一用户 1 秒内只允许 1 次）
  - 服务层：Redis 预扣库存（Lua 原子操作）→ MQ 异步下单 → DB 最终确认
  - 库存不足直接返回，不进入后续流程

## 三、金融级架构设计

### 3.1 整体架构分层

```
┌─────────────────────────────────────────────────────┐
│                    接入层                              │
│  CDN → WAF → API Gateway（限流/鉴权/路由/灰度）        │
├─────────────────────────────────────────────────────┤
│                    业务层                              │
│  ┌──────────┐ ┌──────────┐ ┌──────────┐ ┌────────┐  │
│  │ 交易服务  │ │ 账户服务  │ │ 支付服务  │ │风控服务 │  │
│  └──────────┘ └──────────┘ └──────────┘ └────────┘  │
│  ┌──────────┐ ┌──────────┐ ┌──────────┐ ┌────────┐  │
│  │ 行情服务  │ │ 用户服务  │ │ 清算服务  │ │通知服务 │  │
│  └──────────┘ └──────────┘ └──────────┘ └────────┘  │
├─────────────────────────────────────────────────────┤
│                    中间件层                            │
│  MQ(RocketMQ) | Cache(Redis) | Search(ES) | Job(XXL)│
├─────────────────────────────────────────────────────┤
│                    数据层                              │
│  MySQL(资金) | TiDB(交易) | MongoDB(日志) | HBase(流水)│
├─────────────────────────────────────────────────────┤
│                    基础设施层                           │
│  K8s | Prometheus+Grafana | ELK | SkyWalking | Vault │
└─────────────────────────────────────────────────────┘
```

### 3.2 核心链路设计

**交易核心链路（同步，< 100ms）**
```
下单请求 → 参数校验 → 风控检查（同步）→ 账户校验 → 撮合/下单
         → 冻结资金 → 返回受理结果
         → 异步：成交确认 → 清算 → 资金划转 → 通知
```
- 关键点：
  - 风控必须同步，不能异步（否则风险交易可能已成交）
  - 资金冻结和下单在同一个事务中
  - 成交后的清算可以异步处理
  - 每一步都有对应的补偿/回滚操作

**支付链路设计**
```
支付请求 → 幂等校验（Token）→ 风控 → 创建支付单（本地事务）
         → 调用渠道（银行/三方）→ 等待回调/主动查询
         → 更新支付状态 → 通知业务方
```
- 关键点：
  - 支付单状态机：待支付 → 支付中 → 成功/失败/超时
  - 渠道调用超时：主动查询补偿（定时任务轮询未终态订单）
  - 掉单处理：对账文件 T+1 核对，差异自动/人工处理
  - 渠道路由：多渠道负载均衡 + 故障自动切换

### 3.3 资金安全架构

**双账本体系**
```
┌─────────────┐     ┌─────────────┐
│   业务账本    │     │   会计账本    │
│  （实时更新）  │ ──→ │ （异步记账）   │
│  用户余额     │     │  借贷平衡     │
│  冻结金额     │     │  科目汇总     │
└─────────────┘     └─────────────┘
         ↓                  ↓
    ┌─────────────────────────┐
    │      对账引擎（T+0/T+1）  │
    │  业务账本 vs 会计账本      │
    │  内部账本 vs 渠道账单      │
    └─────────────────────────┘
```

**资金操作原则**
- 先记账后操作：任何资金变动先写流水，再更新余额
- 借贷必相等：每笔交易的借方和贷方金额必须相等
- 余额不能为负：扣款前必须校验余额 >= 扣款金额（SELECT ... FOR UPDATE）
- 流水不可篡改：交易流水表只 INSERT，禁止 UPDATE/DELETE
- 金额精度：使用 BigDecimal，数据库使用 DECIMAL(20,2)，禁止浮点数
- 幂等保障：每笔交易有全局唯一的交易流水号，重复请求返回相同结果

**防资损设计**
- 实时对账：每笔交易完成后异步核对借贷平衡
- 准实时对账：每 5 分钟汇总核对各账户余额
- 日终对账：T+1 与银行/渠道对账文件逐笔核对
- 资损熔断：对账发现差异超过阈值，自动暂停相关业务
- 资金监控大盘：实时展示各账户余额变动、异常交易告警

### 3.4 高可用架构

**多活架构**
- 同城双活（推荐金融场景起步方案）：
  - 两个机房，同城 < 3ms 延迟
  - 数据库：主库在 A 机房，B 机房半同步从库
  - 应用层：两个机房都部署，通过 GSLB 分流
  - 资金类请求：只路由到主库所在机房
  - 非资金请求：就近路由
- 异地多活（高阶方案）：
  - 按用户维度分片，每个用户的数据归属固定单元
  - 单元内自治，跨单元通过消息队列同步
  - 资金操作不跨单元

**容灾切换**
- RTO（恢复时间目标）：核心交易 < 30s，非核心 < 5min
- RPO（恢复点目标）：资金数据 = 0（零丢失），非资金 < 1s
- 切换方式：
  - 自动切换：心跳检测 + 仲裁节点（避免脑裂）
  - 手动切换：提供一键切换工具，定期演练
- 数据库切换：MHA / Orchestrator 自动主从切换
- 应用切换：K8s + Service Mesh 流量切换

**全链路压测**
- 影子库/影子表：压测流量写入影子数据，不污染生产
- 压测标记透传：通过 Header/ThreadLocal 全链路传递
- 压测范围：核心交易链路、支付链路、行情推送
- 压测指标：QPS、RT（P99/P999）、错误率、资源利用率
- 容量规划：日常流量的 3-5 倍作为容量上限

### 3.5 风控架构

```
┌──────────────────────────────────────────┐
│              风控决策引擎                    │
│  ┌────────┐ ┌────────┐ ┌──────────────┐  │
│  │规则引擎 │ │模型引擎 │ │ 实时特征计算   │  │
│  │Drools  │ │PMML   │ │ Flink CEP    │  │
│  └────────┘ └────────┘ └──────────────┘  │
├──────────────────────────────────────────┤
│              风控数据层                     │
│  用户画像 | 设备指纹 | 行为序列 | 关系图谱    │
│  Redis(实时) | HBase(历史) | Neo4j(关系)   │
└──────────────────────────────────────────┘
```

**实时风控（交易前，同步，< 20ms）**
- 规则引擎：黑名单、额度控制、频次控制
- 模型引擎：欺诈评分、信用评分
- 特征计算：最近 1 小时交易次数、金额累计等
- 决策结果：通过 / 拒绝 / 人工审核

**准实时风控（交易后，异步，分钟级）**
- 关联分析：同设备、同 IP、同收款方的交易聚合
- 异常检测：交易金额/频次偏离历史模式
- 团伙识别：图计算发现资金环路

### 3.6 数据架构

**OLTP（在线交易）**
- MySQL：资金账户、核心交易（强一致性）
  - 主从架构，半同步复制
  - 分库分表：用户ID 分 16 库 × 64 表
- TiDB：订单、流水（海量数据 + 在线查询）
  - 兼容 MySQL 协议，水平扩展
  - 适合交易流水这种只增不改的场景

**OLAP（离线分析）**
- ClickHouse：交易报表、风控特征聚合
- Hive/Spark：T+1 批量对账、报表生成
- 数据同步：Canal 监听 MySQL Binlog → Kafka → 下游

**数据生命周期**
- 热数据（< 3 个月）：MySQL/TiDB，SSD 存储
- 温数据（3-12 个月）：TiDB/HBase，SSD/HDD 混合
- 冷数据（> 12 个月）：对象存储（S3/OSS），压缩归档
- 合规要求：交易数据至少保留 5 年，审计日志至少 7 年

## 四、金融合规与监管要求

### 4.1 数据合规
- 用户数据加密存储：AES-256 加密敏感字段
- 传输加密：全链路 TLS 1.2+
- 密钥管理：使用 HSM（硬件安全模块）或 KMS 服务
- 数据脱敏：日志、测试环境、数据分析场景必须脱敏
- GDPR/个人信息保护法：用户数据删除权、可携带权

### 4.2 审计追踪
- 所有资金操作必须有完整审计日志
- 审计日志包含：操作人、操作时间、操作内容、操作前后数据、IP、设备信息
- 审计日志独立存储，业务系统无权修改
- 支持按时间范围、用户、操作类型查询

### 4.3 监管报送
- 大额交易报告（单笔 >= 5 万现金，>= 20 万转账）
- 可疑交易报告（异常模式自动识别 + 人工复核）
- 反洗钱（AML）：客户身份识别（KYC）、交易监控、名单筛查
- 报送数据准确性：与内部数据 100% 一致

## 六、架构审查检查清单

审查金融系统架构或代码时，逐项检查：

### 资金安全（最高优先级）
- [ ] 资金操作是否有事务保障
- [ ] 是否存在资金操作无流水记录的情况
- [ ] 余额变更是否有并发控制（乐观锁/悲观锁）
- [ ] 是否有对账机制覆盖所有资金链路
- [ ] 金额计算是否使用 BigDecimal
- [ ] 是否存在资金操作的幂等漏洞

### 一致性保障
- [ ] CP 场景是否真正做到了强一致
- [ ] AP 场景的最终一致性补偿机制是否完善
- [ ] 分布式事务方案是否有补偿/回滚
- [ ] 缓存与数据库一致性方案是否合理

### 性能与容量
- [ ] 核心链路 RT 是否满足要求（P99）
- [ ] 是否有容量规划和压测数据支撑
- [ ] 数据库是否有慢查询
- [ ] 缓存命中率是否达标（> 95%）
- [ ] 线程池/连接池配置是否合理

### 高可用
- [ ] 核心服务是否有降级方案
- [ ] 外部依赖是否有超时和熔断
- [ ] 是否有容灾切换方案和演练记录
- [ ] 监控告警是否覆盖关键指标

### 安全合规
- [ ] 敏感数据是否加密存储和传输
- [ ] 接口是否有认证授权和防刷
- [ ] 审计日志是否完整
- [ ] 是否满足监管报送要求

